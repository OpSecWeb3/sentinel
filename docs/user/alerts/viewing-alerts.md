# Viewing alerts

Alerts are generated by Sentinel's worker pipeline when a detection rule matches
an incoming event. Alerts are read-only from the dashboard -- they cannot be
manually created or modified. This page explains how to navigate the alert
dashboard, filter alerts, and interpret alert details.

## Navigating the alerts page

Navigate to **Alerts** in the sidebar to open the alert dashboard. The page
displays a command-style header that updates dynamically as you apply filters,
showing the active filter parameters.

Below the header, a summary line shows the total number of alerts and the count
of active filters.

## Alert list columns

The alert list displays a table with the following columns:

| Column | Description |
|---|---|
| **Sev** | The severity level abbreviated as CRIT, HIGH, MED, or LOW. Color-coded: critical in red, high in amber, medium in blue, low in gray. |
| **Title** | A summary of what triggered the alert. Displays "Untitled" if no title was set. |
| **Detection** | The name of the detection that generated the alert. |
| **Module** | The data source module (chain, github, registry, infra, or aws). |
| **Time** | A relative timestamp showing how long ago the alert was created (for example, "5m ago", "2h 30m ago", "3d 12h ago"). Hover over the timestamp to see the full date and time. |
| **Notif** | The notification delivery status, shown as a bracketed label. |

### Notification status labels

| Label | Meaning |
|---|---|
| **[sent]** | All configured notifications were delivered. Displayed in blue. |
| **[pending]** | Alert was created but notifications have not been dispatched yet. Displayed in amber. |
| **[fail]** | All notification attempts failed. Displayed in red. |
| **[partial]** | Some notifications succeeded and some failed. Displayed in amber. |
| **[--]** | The detection has no notification channels configured. |

Select any alert row to open the alert detail view.

## Filtering alerts

Use the filter controls below the search bar to narrow the alert list. Sentinel
provides five filter dropdowns and a text search field.

### Search

Enter a text query in the search bar to search across alert titles and
descriptions. The search is case-insensitive and matches partial strings. Results
update after a short debounce delay.

### Severity filter

Select a severity level to view only alerts at that level.

| Option | Displays |
|---|---|
| CRIT | Critical alerts only |
| HIGH | High alerts only |
| MED | Medium alerts only |
| LOW | Low alerts only |

### Module filter

Filter by the source module to focus on a specific data source. The module list
is populated dynamically from your organization's detections.

### Detection filter

Filter by a specific detection to view all alerts generated by a single
detection. The detection list shows up to 100 detections from your organization.
Detection names longer than 30 characters are truncated in the dropdown.

### Notification status filter

Filter by the delivery state of alert notifications:

| Option | Meaning |
|---|---|
| pending | Notifications not yet dispatched. |
| sent | All notifications delivered. |
| failed | All notification attempts failed. |
| partial | Some notifications succeeded, some failed. |
| no_channels | No notification channels configured on the detection. |

### Time range filter

Select a preset time range to view alerts within a specific window:

| Preset | Duration |
|---|---|
| 1h | Last 1 hour |
| 6h | Last 6 hours |
| 24h | Last 24 hours |
| 7d | Last 7 days |
| 30d | Last 30 days |
| all time | No time restriction (default) |

### Clearing filters

Select the clear button in the filter bar to remove all active filters and reset
the alert list to the default view.

## Pagination

Results are paginated with a default page size of 25. When there are multiple
pages, pagination controls appear at the bottom of the alert list:

- **[<< first]** -- Jump to the first page.
- **[< prev]** -- Go to the previous page.
- **[next >]** -- Go to the next page.
- **[last >>]** -- Jump to the last page.

The current page number and total page count are displayed between the navigation
buttons.

## Sorting

Alerts are sorted by creation time with the most recent alerts displayed first.
This sort order is fixed and cannot be changed from the dashboard.

To review alerts by severity, use the severity filter to isolate a single level,
then browse the results in chronological order.

To review alerts by module, use the module filter.

## Alert detail view

Select an alert row to open the detail view. The detail page shows:

- **Full alert record** -- All fields from the alert, including severity, title,
  description, trigger type, trigger data, notification status, and timestamps.
- **Detection name** -- The name of the detection that generated the alert. Use
  this to navigate to the detection configuration.
- **Detection ID** -- The UUID of the parent detection.
- **Rule ID** -- The specific rule within the detection that matched.
- **Event ID** -- The UUID of the event that triggered the alert.
- **Associated event** -- If the triggering event is available, the full event
  record is loaded and displayed. The event includes the raw payload from the
  source system.

## Understanding alert trigger data

Each alert includes trigger data that contains the raw context from the rule
evaluation. The structure of this data depends on the module and evaluator that
produced the alert.

### Blockchain alerts

Blockchain alerts typically include:

- The contract address and network where the event occurred.
- The decoded event parameters (for example, `from`, `to`, `value` for a Transfer
  event).
- The transaction hash and block number.
- For windowed evaluators, the count or sum that exceeded the threshold.

### GitHub alerts

GitHub alerts typically include:

- The repository name and organization.
- The action that occurred (for example, `publicized`, `created`, `deleted`).
- The actor who performed the action.
- The affected resource (branch, deploy key, member).

### Registry alerts

Registry alerts typically include:

- The artifact name, tag, and digest.
- The type of change detected (digest change, new tag, version published).
- For attribution rules, the CI workflow, actor, and branch information.
- For anomaly detection, the change count and window parameters.

### Infrastructure alerts

Infrastructure alerts typically include:

- The hostname or domain that triggered the alert.
- For certificate alerts, the certificate subject, issuer, and expiry date.
- For DNS alerts, the record type and old/new values.
- For availability alerts, the response time and failure count.

### AWS alerts

AWS alerts typically include:

- The CloudTrail event name and source.
- The principal ID and ARN of the actor.
- The AWS region where the action occurred.
- For error events, the error code and message.

## Alert lifecycle

Alerts are created in real time as events flow through the Sentinel pipeline. The
typical lifecycle is:

1. **Event received.** Sentinel ingests an event from a module (webhook, poller,
   or CloudTrail).
2. **Event normalized.** The module normalizer converts the raw event into
   Sentinel's internal event format.
3. **Rules evaluated.** The rule engine evaluates the event against all active
   detections for the organization.
4. **Alert created.** If one or more rules match and no suppress rules override
   them, Sentinel creates an alert record.
5. **Notification dispatched.** The alert dispatch worker loads the detection's
   channel configuration and sends notifications. Delivery results are recorded as
   notification delivery records.
6. **Alert updated.** The alert's notification status is updated to reflect the
   delivery outcome.

Alerts are immutable once created. The only field that changes after creation is
the notification status, which is updated after the dispatch worker completes.

## Dashboard statistics

The alerts API provides aggregate statistics accessible from the dashboard
overview:

- **Total alerts** -- The all-time count of alerts for your organization.
- **Today** -- Alerts created since midnight (server local time).
- **This week** -- Alerts created in the past 7 days.
- **Active detections** -- The number of detections currently in `active` status.
- **Severity breakdown** -- A summary of alert counts by severity level.
- **Recent alerts** -- The 10 most recent alerts with their severity and detection
  name.
