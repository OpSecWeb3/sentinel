# Viewing alerts

Alerts are generated by Sentinel's worker pipeline when a detection rule matches an incoming event. Alerts are read-only from the dashboard -- they cannot be manually created or modified. This page explains how to navigate the alert dashboard, filter alerts, and interpret alert details.

## Alert dashboard overview

The alert dashboard is your primary interface for reviewing security findings. It displays a paginated list of alerts sorted by creation time (most recent first). Each row shows:

- **Severity** -- The severity level inherited from the detection (`critical`, `high`, `medium`, or `low`).
- **Title** -- A summary of what triggered the alert.
- **Description** -- Additional context about the finding.
- **Detection name** -- The detection that generated the alert, linked to the detection detail page.
- **Module** -- The data source module (chain, github, registry, infra, or aws).
- **Trigger type** -- The specific rule type that fired.
- **Notification status** -- Whether notifications were delivered (`sent`, `partial`, `failed`, `pending`, or `no_channels`).
- **Created at** -- The timestamp when the alert was created.

The dashboard also displays aggregate statistics:

- **Total alerts** -- The all-time count of alerts for your organization.
- **Today** -- Alerts created since midnight (server local time).
- **This week** -- Alerts created in the past 7 days.
- **Active detections** -- The number of detections currently in `active` status.
- **Severity breakdown** -- A summary of alert counts by severity level.
- **Recent alerts** -- The 10 most recent alerts with their severity and detection name.

## Filtering alerts

Use the filter controls at the top of the alert list to narrow results.

**By severity.** Select a severity level to view only alerts at that level. Options: `critical`, `high`, `medium`, `low`.

**By module.** Filter by the source module to focus on a specific data source. When you filter by module, Sentinel uses the detection's `moduleId` to match alerts.

**By detection.** Filter by a specific detection ID to view all alerts generated by a single detection.

**By trigger type.** Filter by the rule type that fired (for example, `chain.event_match` or `registry.digest_change`).

**By notification status.** Filter by the delivery state of alert notifications:

| Status | Meaning |
|---|---|
| `pending` | Alert was created but notifications have not been dispatched yet. |
| `sent` | All configured notifications were delivered. |
| `partial` | Some notifications succeeded and some failed. |
| `failed` | All notification attempts failed. |
| `no_channels` | The detection has no notification channels configured. |

**By time range.** Set a **From** and **To** date-time range to view alerts within a specific window. Dates must be in ISO 8601 format.

**By search.** Enter a text query to search across alert titles and descriptions. The search is case-insensitive and matches partial strings.

**Pagination.** Results are paginated with a default page size of 20 (maximum 100). Use the page controls to navigate through results.

## Alert detail view

Select an alert row to open the detail view. The detail page shows:

- **Full alert record** -- All fields from the alert, including severity, title, description, trigger type, trigger data, notification status, and timestamps.
- **Detection name** -- The name of the detection that generated the alert. Use this to navigate to the detection configuration.
- **Detection ID** -- The UUID of the parent detection.
- **Rule ID** -- The specific rule within the detection that matched.
- **Event ID** -- The UUID of the event that triggered the alert.

## Understanding alert context and trigger data

Each alert includes a `triggerData` field that contains the raw context from the rule evaluation. The structure of this field depends on the module and evaluator that produced the alert.

**Blockchain alerts** typically include:

- The contract address and network where the event occurred.
- The decoded event parameters (for example, `from`, `to`, `value` for a Transfer event).
- The transaction hash and block number.
- For windowed evaluators, the count or sum that exceeded the threshold.

**GitHub alerts** typically include:

- The repository name and organization.
- The action that occurred (for example, `publicized`, `created`, `deleted`).
- The actor who performed the action.
- The affected resource (branch, deploy key, member).

**Registry alerts** typically include:

- The artifact name, tag, and digest.
- The type of change detected (digest change, new tag, version published).
- For attribution rules, the CI workflow, actor, and branch information.
- For anomaly detection, the change count and window parameters.

**Infrastructure alerts** typically include:

- The hostname or domain that triggered the alert.
- For certificate alerts, the certificate subject, issuer, and expiry date.
- For DNS alerts, the record type and old/new values.
- For availability alerts, the response time and failure count.

**AWS alerts** typically include:

- The CloudTrail event name and source.
- The principal ID and ARN of the actor.
- The AWS region where the action occurred.
- For error events, the error code and message.

## Alert timeline

Alerts are created in real time as events flow through the Sentinel pipeline. The typical lifecycle is:

1. **Event received.** Sentinel ingests an event from a module (webhook, poller, or CloudTrail).
2. **Event normalized.** The module normalizer converts the raw event into Sentinel's internal event format.
3. **Rules evaluated.** The rule engine evaluates the event against all active detections for the organization.
4. **Alert created.** If one or more rules match and no suppress rules override them, Sentinel creates an alert record.
5. **Notification dispatched.** The alert dispatch worker loads the detection's channel configuration and sends notifications. Delivery results are recorded as notification delivery records.
6. **Alert updated.** The alert's `notificationStatus` is updated to reflect the delivery outcome.

Alerts are immutable once created. The only field that changes after creation is `notificationStatus`, which is updated after the dispatch worker completes.
