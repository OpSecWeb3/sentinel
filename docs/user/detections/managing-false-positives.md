# Managing false positives

False positives occur when a detection generates an alert for activity that is
benign or expected. A high false-positive rate leads to alert fatigue, which
causes your team to miss real threats. Sentinel provides several mechanisms to
reduce false positives without sacrificing detection coverage.

## Understanding false positives in security monitoring

Every alert in Sentinel is generated by a rule evaluator within a detection. When
the evaluator matches an incoming event, the detection fires. False positives
arise when:

- **Thresholds are too sensitive.** A transfer volume monitor set to 100 tokens
  fires on routine treasury operations.
- **Scope is too broad.** A branch protection monitor watches all branches,
  including temporary feature branches that are expected to have protection
  toggled.
- **Legitimate automation triggers detections.** A CI/CD pipeline creates deploy
  keys or pushes Docker images outside business hours.
- **Cooldowns are too short.** A single incident causes a burst of related
  events, each generating a separate alert.

Identifying the root cause of false positives is the first step toward effective
tuning. Review the alert detail view to inspect the event payload that triggered
each false positive before making changes.

## Tuning detection thresholds

Adjusting rule thresholds is the most direct way to reduce false positives.

1. Navigate to **Detections** and select the detection that is producing false
   positives.
2. Select **[edit]** on the detection detail page.
3. Adjust numeric thresholds in the rule configuration:
   - For **windowed count** rules, increase the `threshold` value to require more
     events before alerting.
   - For **windowed sum** rules, increase the volume `threshold`.
   - For **windowed spike** rules, increase the `increasePercent` value to require
     a larger deviation from baseline.
   - For **balance track** rules, widen the `dropPercent` or adjust the
     `conditionValue`.
4. Consider increasing the **time window** (`windowMinutes`). A wider window
   smooths out short bursts of legitimate activity.
5. Save the detection. Sentinel rebuilds the rules with your updated parameters.

**Tip:** Before changing thresholds, review the alert history on the **Alerts**
page. Filter by the detection name to see recent triggers and compare the event
payload values against your expected baseline.

## Using exclude patterns

Many evaluators support filters that restrict matching to specific resources. Use
these to exclude known-good activity from detection scope.

### Blockchain detections

- Specify a single contract address instead of monitoring all contracts on a
  network. On the detection edit form, select a specific contract from the
  dropdown.
- Use `conditions` arrays in event match rules to filter on decoded parameter
  values. For example, add a condition that filters transfers by a specific sender
  address to alert only on transfers from a particular entity.
- Use the `groupByField` parameter to count or sum per entity, so a single
  high-activity address does not skew aggregate thresholds.

### GitHub detections

- Use `excludeRepos` patterns to ignore repositories where visibility changes are
  expected (for example, `org/archived-*`).
- Use `watchBranches` patterns to limit branch protection and force push
  monitoring to critical branches only (for example, `main`, `release/*`).
- Use `watchRoles` to narrow member change monitoring to specific roles (for
  example, `owner`).

### Registry detections

- Use `tagPatterns` to match only specific image tags or npm dist-tags instead of
  the wildcard `*`. For example, monitor only `latest` and `production` tags.
- Use `pusherAllowlist` in anomaly detection rules to suppress alerts for known CI
  bot users.
- Use `workflows`, `actors`, and `branches` in attribution rules to define your
  expected CI pipeline.

### AWS detections

- Use `excludeEventNames` to suppress expected root account activity (for example,
  `GetBillingDetails`).
- Use `principalArnPatterns` to restrict monitoring to specific IAM roles or
  users.
- Use `eventSources` to limit event matching to specific AWS services.

### Infrastructure detections

- Configure the **host scope** field on infrastructure detections to limit
  monitoring to specific hostnames or glob patterns.
- Configure `ignorePatterns` on subdomain and CT log monitors to suppress alerts
  for known subdomains.
- Set `requiredHeaders` to only alert on the specific security headers your policy
  mandates.

## Adjusting time windows

Increasing time windows reduces false positives caused by short bursts of
legitimate activity.

For **windowed count**, **windowed sum**, and **windowed spike** evaluators, the
`windowMinutes` parameter defines the rolling time window for aggregation.

- A 5-minute window is sensitive to short bursts and may fire on batch operations.
- A 30-minute window smooths out routine activity and fires only on sustained
  anomalies.
- A 60-minute window provides the broadest view and is appropriate for
  low-frequency monitoring.

To adjust the time window:

1. Navigate to the detection detail page and select **[edit]**.
2. Locate the time window input in the rule configuration.
3. Increase the value to broaden the aggregation window.
4. Save the detection.

Keep in mind that wider windows increase the latency between the start of an
attack and the first alert. Balance sensitivity against your response time
requirements.

## Disabling noisy rules

When a detection consistently produces unacceptable false-positive rates after
tuning, consider pausing or archiving it.

### Pausing a detection

Pausing suspends evaluation without deleting the detection or its alert history.

1. Navigate to the **Detections** list page.
2. Locate the detection and select **[pause]** in the actions column. You can also
   pause from the detection detail page.
3. The detection status changes to **paused**. All of its rules are paused
   simultaneously.
4. To reactivate later, select **[resume]**.

Use pausing when you plan to return to the detection after gathering more data
about expected event patterns.

### Archiving a detection

Archiving permanently disables a detection and removes it from the default list
view.

1. Navigate to the **Detections** list page.
2. Select **[archive]** in the actions column. Sentinel displays a confirmation
   dialog.
3. Confirm the archive action. The detection status changes to **archived** and
   the detection is hidden from the default list.
4. To view archived detections, filter the detection list by the **archived**
   status.

Archived detections cannot be edited or reactivated. Their alert history is
preserved for audit purposes.

### Using suppress rules instead of disabling

If only certain patterns produce false positives, add a `suppress` rule instead
of disabling the entire detection. This preserves coverage for legitimate threats
while silencing known-good activity.

1. Navigate to the detection detail page and select **[edit]**.
2. Add a new rule with the action set to `suppress`.
3. Configure the rule to match the specific pattern you want to ignore. For
   example:
   - A `chain.event_match` suppress rule matching transfers from your known
     treasury address.
   - A `registry.anomaly_detection` suppress rule matching pushes from
     `github-actions[bot]`.
4. Set the suppress rule's **priority** to a lower number (higher precedence) than
   your alert rules. This ensures the suppress rule evaluates first.
5. Save the detection.

When a suppress rule matches an incoming event, the detection does not fire --
even if alert rules also match the same event.

**Example: Allowlisting a known deployer**

You have a "Contract Creation Watcher" detection that fires when any monitored
address deploys a contract. Your CI pipeline deploys contracts from a known
deployer address. To suppress alerts for this address:

- Add a suppress rule with `ruleType: chain.event_match`, config containing the
  deployer address and `matchContractCreation: true`, action `suppress`, and
  priority `10`.
- Keep the existing alert rule at priority `50`.

## Best practices for rule tuning

### Establish a regular review cadence

Schedule recurring triage sessions (weekly or biweekly) to review detections with
high alert volumes.

1. **Identify repeat offenders.** On the **Alerts** page, filter by detection name
   and review the alert volume over the past week. Detections with a high
   alert-to-action ratio are candidates for tuning.

2. **Inspect alert payloads.** Open several recent alerts from the same detection
   and review the trigger data. Identify the common characteristics of
   false-positive events (specific addresses, usernames, event types, or time
   patterns).

3. **Choose a tuning strategy.** Based on your findings, select one or more of the
   approaches described above:
   - Raise thresholds if the events are legitimate but below your risk tolerance.
   - Add resource filters if the events come from known-good sources.
   - Add suppress rules if you need to allowlist specific patterns while keeping
     the detection active for other patterns.
   - Increase the cooldown if the detection fires too frequently on the same
     underlying incident.

4. **Apply changes and monitor.** After tuning, monitor the detection for one to
   two weeks. Compare the alert volume before and after the change.

5. **Document your tuning decisions.** Use the detection **description** field to
   record why you changed specific thresholds or added suppress rules. This helps
   other team members understand the detection's history and avoids undoing
   previous tuning work.

### Start permissive, then tighten

When deploying a new detection, start with relaxed thresholds and the `log` rule
action. Monitor the logged events for a baseline period, then switch to `alert`
and tighten thresholds based on observed data.

### Use cooldowns strategically

Cooldowns do not suppress detection evaluation -- events are still matched and
logged. They only prevent redundant alert creation and notification dispatch. Use
cooldowns to manage notification volume without losing visibility into event
patterns.

| Scenario | Recommended cooldown |
|---|---|
| High-frequency blockchain event monitors | 15 to 60 minutes |
| Infrastructure probes that may flap | 30 to 60 minutes |
| GitHub organization monitors during planned changes | 60 to 120 minutes |
| Rare critical events (proxy upgrades, root account activity) | 0 minutes |

### Avoid over-tuning

Removing too many patterns can create blind spots. When adding suppress rules or
narrowing scope, document what you are excluding and why. Periodically revisit
your suppression rules to ensure they remain appropriate as your environment
changes.
