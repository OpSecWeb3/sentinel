# Managing false positives

False positives occur when a detection generates an alert for activity that is benign or expected. A high false-positive rate leads to alert fatigue, which causes your team to miss real threats. Sentinel provides several mechanisms to reduce false positives without sacrificing detection coverage.

## Understanding false positives in Sentinel

Every alert in Sentinel is generated by a rule evaluator within a detection. When the evaluator matches an incoming event, the detection fires. False positives arise when:

- **Thresholds are too sensitive.** A transfer volume monitor set to 100 tokens fires on routine treasury operations.
- **Scope is too broad.** A branch protection monitor watches all branches, including temporary feature branches that are expected to have protection toggled.
- **Legitimate automation triggers detections.** A CI/CD pipeline creates deploy keys or pushes Docker images outside business hours.
- **Cooldowns are too short.** A single incident causes a burst of related events, each generating a separate alert.

## Tuning detection sensitivity

Adjusting rule thresholds is the most direct way to reduce false positives.

1. Navigate to **Detections** and select the detection that is producing false positives.
2. Select **Edit Detection**.
3. Adjust numeric thresholds in the rule configuration:
   - For **windowed count** rules, increase the `threshold` value to require more events before alerting.
   - For **windowed sum** rules, increase the volume `threshold`.
   - For **windowed spike** rules, increase the `increasePercent` value to require a larger deviation from baseline.
   - For **balance track** rules, widen the `dropPercent` or adjust the `conditionValue`.
4. Consider increasing the **time window** (`windowMinutes`). A wider window smooths out short bursts of legitimate activity.
5. Save the detection. Sentinel rebuilds the rules with your updated parameters.

**Tip:** Before changing thresholds, review the alert history to understand the typical values that trigger false positives. Use the alert detail view to inspect the event payload and compare it against your expected baseline.

## Using resource filters to narrow scope

Many evaluators support filters that restrict matching to specific resources.

**Blockchain detections:**

- Specify a single contract address instead of monitoring all contracts on a network.
- Use `conditions` arrays in event match rules to filter on decoded parameter values. For example, add a condition that filters transfers by a specific sender address to alert only on transfers from a particular entity.
- Use the `groupByField` parameter to count or sum per entity, so a single high-activity address does not skew aggregate thresholds.

**GitHub detections:**

- Use `excludeRepos` patterns to ignore repositories where visibility changes are expected (for example, `org/archived-*`).
- Use `watchBranches` patterns to limit branch protection and force push monitoring to critical branches only (for example, `main`, `release/*`).
- Use `watchRoles` to narrow member change monitoring to specific roles (for example, `owner`).

**Registry detections:**

- Use `tagPatterns` to match only specific image tags or npm dist-tags instead of the wildcard `*`. For example, monitor only `latest` and `production` tags.
- Use `pusherAllowlist` in anomaly detection rules to suppress alerts for known CI bot users.
- Use `workflows`, `actors`, and `branches` in attribution rules to define your expected CI pipeline.

**AWS detections:**

- Use `excludeEventNames` to suppress expected root account activity (for example, `GetBillingDetails`).
- Use `principalArnPatterns` to restrict monitoring to specific IAM roles or users.
- Use `eventSources` to limit event matching to specific AWS services.

**Infrastructure detections:**

- Configure `ignorePatterns` on subdomain and CT log monitors to suppress alerts for known subdomains.
- Set `requiredHeaders` to only alert on the specific security headers your policy mandates.

## Adjusting cooldown periods

Cooldown periods suppress duplicate alerts from the same detection within a defined time window. Increasing the cooldown reduces the total number of alerts without disabling the detection.

1. Navigate to the detection detail page.
2. Select **Edit Detection**.
3. Increase the **Cooldown** value. Recommended starting points:
   - **15 minutes** for high-frequency blockchain event monitors.
   - **60 minutes** for infrastructure probes that may flap.
   - **120 minutes** for GitHub organization monitors during periods of planned changes.
4. Save the detection.

The cooldown timer resets each time the detection fires. During the cooldown window, matching events are still evaluated and logged, but no new alert is created and no notification is sent.

## Suppressing specific patterns

Use `suppress` rules within a detection to prevent alerts for known-good patterns.

1. Navigate to the detection detail page and select **Edit Detection**.
2. Add a new rule with the action set to `suppress`.
3. Configure the rule to match the specific pattern you want to ignore. For example:
   - A `chain.event_match` suppress rule matching transfers from your known treasury address.
   - A `registry.anomaly_detection` suppress rule matching pushes from `github-actions[bot]`.
4. Set the suppress rule's **priority** to a lower number (higher precedence) than your alert rules. This ensures the suppress rule evaluates first.
5. Save the detection.

When a suppress rule matches an incoming event, the detection does not fire -- even if alert rules also match the same event.

**Example: Allowlisting a known deployer**

You have a "Contract Creation Watcher" detection that fires when any monitored address deploys a contract. Your CI pipeline deploys contracts from a known deployer address. To suppress alerts for this address:

- Add a suppress rule with `ruleType: chain.event_match`, config containing the deployer address and `matchContractCreation: true`, action `suppress`, and priority `10`.
- Keep the existing alert rule at priority `50`.

## Review workflow for recurring false positives

Establish a regular triage process to address detections that produce recurring false positives.

1. **Identify repeat offenders.** Filter the alert dashboard by detection name and review the alert volume over the past week. Detections with a high alert-to-action ratio are candidates for tuning.

2. **Inspect alert payloads.** Open several recent alerts from the same detection and review the trigger data. Identify the common characteristics of false-positive events (specific addresses, usernames, event types, or time patterns).

3. **Choose a tuning strategy.** Based on your findings, select one or more of the approaches described above:
   - Raise thresholds if the events are legitimate but below your risk tolerance.
   - Add resource filters if the events come from known-good sources.
   - Add suppress rules if you need to allowlist specific patterns while keeping the detection active for other patterns.
   - Increase the cooldown if the detection fires too frequently on the same underlying incident.

4. **Apply changes and monitor.** After tuning, monitor the detection for one to two weeks. Compare the alert volume before and after the change.

5. **Pause if necessary.** If a detection continues to produce unacceptable false-positive rates after tuning, pause it temporarily by setting the status to `paused`. Pausing a detection also pauses all of its rules. You can reactivate it later when you have more information about the expected event patterns.

6. **Document your tuning decisions.** Use the detection description field to record why you changed specific thresholds or added suppress rules. This helps other team members understand the detection's history and avoids undoing previous tuning work.
